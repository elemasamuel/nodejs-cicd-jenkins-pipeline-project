# NodeJS Deployment to AWS EC2 with Jenkins

This is a step-by-step walkthrough of how I took my NodeJS application, already wrapped in a CI/CD pipeline, and deployed it to a self-managed AWS EC2 server. By the end, every push I make to the main branch on GitHub builds a new Docker image, pushes it to DockerHub, deploys it automatically to my EC2 server, and the app is reachable in a browser. Feature branches just run tests.

## Scenario

My company decided to move everything to AWS instead of juggling multiple cloud providers. In the previous exercises, I set up a NodeJS app with Jenkins, Docker, and a CI pipeline running on DigitalOcean. In this exercise, I moved the actual server hosting the app onto an AWS EC2 instance, set up entirely through the AWS CLI, and wired my existing pipeline to deploy to it automatically.

## Tech stack

- **AWS CLI** for creating and configuring all AWS resources
- **EC2** for the server hosting the app
- **VPC, subnets, security groups** for the network the server lives in
- **Docker** and **Docker Compose** for running the app on the server
- **Jenkins** for the CI/CD pipeline (in my case, also hosted on the same EC2 instance)
- **GitHub webhooks** for automatic triggering on every push

## Step 1: Create an IAM user

Logging in as root for everyday work is bad practice, so the first thing I did was create a dedicated user with only the permissions I needed for this project.

In the AWS Console:

1. I went to **IAM → Users → Add users** and created a user (I called mine `sam`)
2. I enabled console access with an autogenerated password
3. I created a new group, `devops`, and attached the **AmazonEC2FullAccess** and **AmazonVPCFullAccess** policies to it
4. I added the user to that group
5. I downloaded the `.csv` file with the login URL and temporary password

I logged in as the new user once to set a real password, then logged back in as root to generate an access key for the CLI:

1. I went to the user's **Security credentials** tab
2. I created an access key for **Command line interface (CLI)**
3. I downloaded the `.csv` file with the access key ID and secret access key, since it's only shown once

## Step 2: Configure the AWS CLI

```sh
aws configure
  AWS Access Key ID [None]: <access key from the csv>
  AWS Secret Access Key [None]: <secret key from the csv>
  Default region name [None]: eu-central-1
  Default output format [None]: json
```

This told the CLI which account and region to talk to from then on.

## Step 3: Create a VPC

Instead of using my AWS account's default network, I created a dedicated VPC, so I'd have full control over the network my server sits in.

```sh
# create the VPC
aws ec2 create-vpc \
  --cidr-block 10.0.0.0/24 \
  --query Vpc.VpcId \
  --output text

# create a subnet inside it
aws ec2 create-subnet \
  --vpc-id <vpc-id> \
  --cidr-block 10.0.0.32/28 \
  --availability-zone eu-central-1a \
  --query Subnet.SubnetId \
  --output text

# make sure servers in this subnet get a public IP automatically
aws ec2 modify-subnet-attribute \
  --map-public-ip-on-launch \
  --subnet-id <subnet-id>
```

Then I gave the subnet a path to and from the internet:

```sh
# create an internet gateway and attach it to the VPC
aws ec2 create-internet-gateway --query InternetGateway.InternetGatewayId --output text
aws ec2 attach-internet-gateway --vpc-id <vpc-id> --internet-gateway-id <gateway-id>

# create a route table, add a route to the internet, and associate it with the subnet
aws ec2 create-route-table --vpc-id <vpc-id> --query RouteTable.RouteTableId --output text
aws ec2 create-route --route-table-id <route-table-id> --destination-cidr-block 0.0.0.0/0 --gateway-id <gateway-id>
aws ec2 associate-route-table --route-table-id <route-table-id> --subnet-id <subnet-id>
```

Finally, I created a security group to act as the firewall:

```sh
aws ec2 create-security-group \
  --group-name ssh-access \
  --description "Security group for SSH access" \
  --vpc-id <vpc-id> \
  --query GroupId \
  --output text

# allow SSH access (port 22)
aws ec2 authorize-security-group-ingress \
  --group-id <security-group-id> \
  --protocol tcp \
  --port 22 \
  --cidr <your-ip>/32
```

Since I was on a VPN with a frequently changing IP, I ended up opening SSH to all IPs (`0.0.0.0/0`) instead of locking it to a single address. The key file is still required to actually log in, so this felt like a reasonable tradeoff for a learning project.

## Step 4: Create the EC2 instance

```sh
# create an SSH key pair and save the private key locally
aws ec2 create-key-pair \
  --key-name WebServerKeyPair \
  --query "KeyMaterial" \
  --output text > WebServerKeyPair.pem

chmod 400 WebServerKeyPair.pem

# launch the instance
aws ec2 run-instances \
  --image-id <amazon-linux-ami-id> \
  --count 1 \
  --instance-type t3.micro \
  --key-name WebServerKeyPair \
  --security-group-ids <security-group-id> \
  --subnet-id <subnet-id> \
  --associate-public-ip-address \
  --query "Instances[].InstanceId" \
  --output text

# check it's running and grab its public IP
aws ec2 describe-instances \
  --instance-id <instance-id> \
  --query "Reservations[].Instances[].{State:State.Name,Address:PublicIpAddress}"
```

A couple of things I ran into here. The AMI ID isn't universal, it's tied to the region, so I had to look mine up rather than reuse one from an example. My account's free tier also turned out not to cover `t2.micro`, I had to switch to `t3.micro` instead, which is the modern equivalent.

## Step 5: Install Docker on the server

```sh
ssh -i WebServerKeyPair.pem ec2-user@<public-ip>
```

Then on the server:

```sh
sudo yum update -y
sudo yum install -y docker
sudo systemctl start docker
sudo usermod -aG docker ec2-user

sudo curl -SL https://github.com/docker/compose/releases/download/v2.17.2/docker-compose-linux-x86_64 -o /usr/local/bin/docker-compose
sudo chmod +x /usr/local/bin/docker-compose

docker-compose --version
```

## Step 7: Host Jenkins on the same EC2 server

Rather than spinning up a separate server just for Jenkins, I decided to run it directly on the EC2 instance alongside the app.

```sh
# open Jenkins's web port
aws ec2 authorize-security-group-ingress \
  --group-id <security-group-id> \
  --protocol tcp \
  --port 8080 \
  --cidr 0.0.0.0/0
```

On the server, I added the Jenkins package repo first:

```sh
sudo wget -O /etc/yum.repos.d/jenkins.repo https://pkg.jenkins.io/redhat-stable/jenkins.repo
sudo rpm --import https://pkg.jenkins.io/redhat-stable/jenkins.io-2023.key
```

The latest Jenkins release requires Java 21, which Amazon Linux's default repos don't offer (they top out at Java 17). Instead of chasing Java 21 down through a third-party repo, I installed a Jenkins LTS version that still supports Java 17:

```sh
sudo yum install -y java-17-amazon-corretto
sudo yum install -y https://get.jenkins.io/redhat-stable/jenkins-2.426.1-1.1.noarch.rpm
```

I learned the hard way that Jenkins's built-in disk space monitor checks `/tmp`, which is a small partition by default on this instance type, often under 500MB. Since that's smaller than Jenkins's 1GB threshold, my build node kept getting marked offline before the pipeline could even run. I resized it before starting Jenkins this time, so I wouldn't hit that again:

```sh
sudo mount -o remount,size=2G /tmp
```

My pipeline also runs shell commands against `git`, `node`, and `npm` directly on this server, and runs `docker build`/`docker push` as the Jenkins system user, so I made sure all of that was in place before my first build:

```sh
sudo yum install -y git

curl -fsSL https://rpm.nodesource.com/setup_18.x | sudo bash -
sudo yum install -y nodejs

sudo usermod -aG docker jenkins
```

Then I started Jenkins:

```sh
sudo systemctl start jenkins
sudo systemctl enable jenkins
```

Since I'd just added the `jenkins` user to the `docker` group, I restarted the service once so that membership would actually take effect:

```sh
sudo systemctl restart jenkins
```

Once Jenkins was up, I grabbed the initial admin password and finished the setup wizard in the browser:

```sh
sudo cat /var/lib/jenkins/secrets/initialAdminPassword
```

```
http://<public-ip>:8080
```

## Step 8: Reconnect my pipeline

Since this was a fresh Jenkins instance, I had to set up the credentials, shared library, and pipeline job from my earlier exercises again.

- I added an SSH credential (ID `ec2-server-key`) with my EC2 key pair, so Jenkins could connect to the server
- I added a DockerHub credential (ID `DockerHub`) and a GitHub credential (ID `GitHub`, using a personal access token)
- I registered my shared library under **Manage Jenkins → System → Global Pipeline Libraries**
- I recreated my pipeline job, pointing it at my project's GitHub repo
- I installed the plugins my pipeline depends on: **Pipeline Utility Steps** (for `readJSON`) and **SSH Agent** (for `sshagent`)

My Jenkinsfile's deploy stage copies the compose file to the server and starts the container:

```groovy
stage('Deploy to EC2') {
    when {
        expression {
            params.deploy
        }
    }
    steps {
        script {
            echo 'deploying Docker image to EC2 server...'

            def dockerComposeCmd = "IMAGE_TAG=${IMAGE_VERSION} docker-compose up -d"
            def ec2Instance = "ec2-user@<public-ip>"

            sshagent(['ec2-server-key']) {
                sh "scp -o StrictHostKeyChecking=no docker-compose.yaml ${ec2Instance}:/home/ec2-user"
                sh "ssh -o StrictHostKeyChecking=no ${ec2Instance} ${dockerComposeCmd}"
            }
        }
    }
}
```

## Step 10: Restrict deployment to the main branch, and trigger automatically

I wanted feature branches to only run tests, and only `main` to bump the version, build and push the image, deploy, and commit the version bump back.

```groovy
#!/usr/bin/env groovy
library('jenkins-shared-library')
pipeline {
    agent any
    parameters {
        booleanParam(name: 'deploy', defaultValue: true, description: 'Deploy the application on the EC2 server.')
    }
    stages {
        stage('Bump Version') {
            when {
                expression {
                    return env.GIT_BRANCH == "origin/main"
                }
            }
            steps {
                script {
                    bumpNpmVersion('app', 'patch')
                }
            }
        }
        stage('Run Tests') {
            steps {
                script {
                    runNpmTests('app')
                }
            }
        }
        stage('Build and Push Docker Image') {
            when {
                expression {
                    return env.GIT_BRANCH == "origin/main"
                }
            }
            steps {
                script {
                    buildAndPublishImage("elemasamuel/sam-repo:nodejs-cicd-jenkins-pipeline-project-${IMAGE_VERSION}")
                }
            }
        }
        stage('Deploy to EC2') {
            when {
                expression {
                    return env.GIT_BRANCH == "origin/main" && params.deploy
                }
            }
            steps {
                script {
                    echo 'deploying Docker image to EC2 server...'
                    def dockerComposeCmd = "IMAGE_TAG=${IMAGE_VERSION} docker-compose up -d"
                    def ec2Instance = "ec2-user@<public-ip>"
                    sshagent(['ec2-server-key']) {
                        sh "scp -o StrictHostKeyChecking=no docker-compose.yaml ${ec2Instance}:/home/ec2-user"
                        sh "ssh -o StrictHostKeyChecking=no ${ec2Instance} ${dockerComposeCmd}"
                    }
                }
            }
        }
        stage('Commit Version Update') {
            when {
                expression {
                    return env.GIT_BRANCH == "origin/main"
                }
            }
            steps {
                script {
                    commitAndPushVersionUpdate('github.com/elemasamuel/nodejs-cicd-jenkins-pipeline-project.git', 'GitHub', 'main')
                }
            }
        }
    }
}
```

Then I wired up the webhook so pushes would trigger Jenkins automatically, instead of me clicking **Build Now** every time:

1. On my GitHub repo, I went to **Settings → Webhooks → Add webhook**
2. **Payload URL**: `http://<public-ip>:8080/github-webhook/`
3. I saved it

In Jenkins, I opened the pipeline job's configuration, scrolled to **Build Triggers**, and checked **GitHub hook trigger for GITScm polling**.

To stop Jenkins from triggering a build on its own automatic version-bump commits, I added an extra Git behavior under **Additional Behaviours**: **Polling ignores commits from certain users**, with the username `jenkins`.

---

## A note on the commit-back stage

My "Commit Version Update" stage pushes the version bump in `package.json` back to GitHub after every build. Because my Jenkins workspace and GitHub's history drifted apart over many builds, especially after re-running the pipeline several times while debugging, this push kept getting rejected. Since the stage only ever touches that one file and nothing else relies on a clean merge history, I fixed it by pushing with `--force` in my shared library function:

```groovy
sh "git push --force origin HEAD:${branch}"
```
